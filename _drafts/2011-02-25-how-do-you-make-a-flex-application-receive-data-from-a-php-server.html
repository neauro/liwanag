The other side of that one project that I'm working on for PhotoCity.

Apparently PHP and Flex can work pretty well together, but all I'm interested in now is how a PHP page can send back data to a Flex application to work with. In particular, all I want to do for now is for Flex to take a simple line of data (the x, y, and z coordinates of a point) that PHP returns to it.

The main issue with sending data from PHP to Flex is how to structure the data. One way that PHP can communicate data back to Flex is by formatting output in XML, which Flex can then read and put into pretty charts, but I don't want to worry too much about formatting in XML yet.

Looking around, I found that the Zend Framework is another amazing way that you can make an application with a Flex front-end and a PHP(5) back-end, with which you can easily return a database table through PHP to Flex in an array. But this still seems a little bit more than the simple functionality I want.

Something else promising: using Flex's RemoteObject class, which can be used by a Flex application as if it were a remote PHP class. (Remember remote procedure calls?!) Just like other UI components in Flex, you can make a RemoteObject components, which will represent an HTTPService object, and allow you to access specific functions of a remote object. For instance, if you define a class data in PHP, with a function getData, in Flex you could call data.getData, and that will be like calling the function getData.


(source)

However, PHP doesn't support remoting natively, you you'd need a server-side library to enable it: thus, Zend again (which is Adobe's recommendation), or AMFPHP, etc. (You can use RemoteObject to call methods directly on a ColdFusion component or a Java class.)

Additionally, you could facilitate communication between Flex and PHP by using PHP as a socket server and connecting to it with a Flex application. The advantage of using sockets to send data from a server to a client is that, unlike with the HTTP client/server communication model, which needs a lot of exta information (i.e. to support how the client sends a request to server, then server processes the request, then client receives a response and disconnects), with a socket, the application makes a single request, and the server opens a connection and maintains it so that it can push data out to the client whenever it feels like it. This also seems kind of complicated, as Flash Player requires policy files to be on the server the client is connecting to, and then you need a socket object...
